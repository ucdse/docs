# Frontend

During the early engineering stage of the **Dublin Bike Usage Prediction** project's frontend, I was initially unsure about which React project scaffolding to choose—the traditional `create-react-app` had ceased maintenance, and it remained uncertain whether newer build tools like Vite were suitable for production-grade projects. In my discussions with AI, it helped me systematically compare `Vite` versus `create-react-app` in terms of build speed, hot module replacement mechanisms, and React 19 compatibility, ultimately recommending that I scaffold the project using the `npm create vite@latest` command with `React 19` and `TypeScript`. Building on this foundation, AI also guided me through the core concepts of React's functional component and Hooks paradigm—from the basic usage of `useState` and `useEffect` to the global state propagation of `useContext`, and further to the declarative data flow of `React Router v6`'s `loader` and `action` in route management. This learning path, progressively structured by AI, enabled me to quickly transition my mindset from class-based components to functional components.

During the development of the **map interaction page**, implementing the multi-modal route navigation query became a bottleneck at one point. I needed to simultaneously handle the place autocomplete of `google.maps.places.Autocomplete`, the concurrent rendering of a walk-cycle-walk three-segment multi-color polyline, and the intelligent zooming of the viewport. AI assisted me in constructing a responsive loop based on Autocomplete, clarifying the hierarchical relationships of frontend state management from a TypeScript type perspective, and helping me implement the overlay rendering of three differently colored polyline segments along the same route, while encapsulating an auto-zoom logic for the `fitBounds` method so that the map viewport could intelligently center on all waypoints.

During the development of the **AI chat page**, I initially adopted the browser's native `EventSource` API to implement server-pushed streaming responses, but quickly discovered that `EventSource` does not support carrying JWT Tokens in request headers, which completely conflicted with the project's security mechanism of uniformly injecting authentication tokens via `axios` interceptors. After I described this compatibility dilemma to the AI, it provided a deep refactoring solution: introducing the `@microsoft/fetch-event-source` library to replace the native `EventSource`, rebuilding the network layer in `api/chat.ts` via `fetchEventSource` so that it could custom-inject the `Authorization: Bearer` token in request headers, while combining it with the `getToken()` function to achieve dynamic token refresh. This solution flawlessly implemented Server-Sent Events (SSE) typewriter-style word-by-word streaming output while maintaining identity authentication. On this basis, combined with the AI-suggested local state interception strategy, I also implemented session state persistence across page refreshes.

# Backend

In the design and implementation of the backend architecture, the first challenge I faced was the Flask application's entry point startup design. The project uses `run.py` as the development environment startup entry, calling `create_app()` to create an application instance and running it with `debug=True`; production deployment, on the other hand, relies on `wsgi.py` as the WSGI server's entry point, which similarly calls `create_app()` but strips away the debug mode. AI helped me clarify the separation of responsibilities between these two entry points—`run.py` is oriented toward hot-reloading during development, while `wsgi.py` conforms to the standard calling conventions of production servers like Gunicorn.

In setting up the Flask application factory pattern, AI helped me organize the structure of the `create_app()` factory function, guiding me to centralize the initialization of extensions (`SQLAlchemy`, `Flask-Migrate`, `Flask-Mail`, `JWT`, etc.) in `app/__init__.py` for unified management, allowing the application to flexibly create instances based on different configuration environments. In particular, when integrating the core **machine learning prediction workflow** with the database, I encountered a severe cold start latency problem—my initial "lazy loading" design caused the backend to dynamically load a massive amount of data into the model for inference during the first API request, resulting in extremely long response times. After analyzing my deployment environment, AI proposed an architectural refactoring solution that leverages the Flask App context lifecycle to perform "pre-warming" at application startup: in `prediction_service.py`, the model and feature data are loaded into memory ahead of time, and through a `copy-on-write` mechanism, redundant memory copying is avoided at the memory level. After this optimization, the backend was able to instantaneously feed prediction features into the model for inference upon receiving a request, returning results to the frontend without any stutter.

In terms of **database table design**, AI assisted me in completing the full modeling process from requirements to SQLAlchemy Models. The `Station` table uses the station number `number` as its primary key, including fields such as `name`, `address`, `position` (latitude and longitude), `banking` (whether a payment terminal is available), and `bonus`; the `Availability` table associates with `station.number` via a foreign key, recording real-time available bike counts and empty parking slot counts; the `Weather` table stores weather observation data; the `User` table adopts an encrypted storage scheme with `email_hash` and `password_hash`, and sets a `verified` field to coordinate with the email verification process; the `Session` table and `ChatHistory` table support the persistent conversation mechanism of LangChain's `SQLChatMessageHistory`. Throughout the design of each table, AI prompted me to pay attention to foreign key constraints, indexing strategies, and field type selections, sparing me from the foreign key association failures caused by type mismatches in the `station.number` field that had occurred in my early design.

Regarding the choice of **database migration** approach, I initially leaned toward manually writing SQL scripts to manage schema changes, but after AI helped me analyze the automated migration workflow of `Flask-Migrate` (based on Alembic), I switched to this approach instead. AI provided detailed guidance on how to execute `flask db init` to initialize the migration directory, `flask db migrate -m "description"` to auto-generate migration scripts, `flask db upgrade` to apply changes to the database, and how to manage and review the auto-generated version files within the `migrations/` directory. This choice allowed the team to avoid the version drift issues that manual SQL scripts would have introduced during collaborative development.

In terms of **LLM conversation integration**, AI helped me complete the integration of LangChain with the Alibaba Cloud Tongyi Qianwen (`qwen-plus`) model in `chat_service.py`, implementing streaming SSE output. Meanwhile, the custom Token generation and verification logic of the `JWT` authentication module, as well as the email verification process implemented by `Flask-Mail`, were also completed under AI's guidance—from the configuration of asynchronous email sending to the secure signing of activation links, AI helped me resolve various obstacles in development one by one.

# Scraper

In the design and implementation of the data scraper, AI assisted me in constructing a scheduled architecture based on dual-thread daemon processes. In `main_scraper.py`, I used `threading.Thread(daemon=True)` to create two persistent background threads: the station thread pulls Dublin bike station real-time data from the `JCDecaux API` every 5 minutes, while the weather thread fetches Dublin weather observation information from the `OpenWeatherMap API` every 1 hour. AI helped me clarify the concurrent safety issues when both threads share database connections, guiding me in designing a reasonable thread scheduling strategy that avoids resource contention caused by overlapping API requests.

In terms of **database communication**, since the Scraper operates as a component independent of the Flask application, it needs to maintain its own database connection. AI guided me in setting up an independent `SQLAlchemy` engine in `database.py`, completely decoupled from the Flask application's `db` instance, while ensuring that the ORM model definitions (`models.py`) remain consistent on both sides. For session management, AI helped me implement a strict `commit/rollback` exception handling flow—after each data write, an explicit commit is ensured; if any exception occurs, an immediate rollback is performed and the error is logged, preventing dirty data from being written to the database.

Regarding **fault tolerance and retry mechanisms**, my initial handling of API request failures was overly simplistic, merely performing basic exception catching. AI pointed out that in the face of network jitter and API rate limiting, this design would lead to data gaps. It assisted me in adding exponential backoff retry logic to each API call, combined with fine-grained judgment of HTTP status codes (distinguishing retriable 5xx errors from non-retriable 4xx client errors), enabling the Scraper to automatically recover when facing transient network failures or API throttling, thereby ensuring the continuous and stable operation of the data collection pipeline.

# Report

During the drafting of the **project report**, AI played a substantial role in both **Markdown formatting** and **document-to-code consistency verification**. On the formatting side, I initially produced the report as plain prose mixed with inconsistent heading levels, ad-hoc bullet indentation, and code snippets that were not fenced properly. AI helped me normalize `project-report-process.md` into a disciplined Markdown structure: standardizing the heading hierarchy (`#`, `##`, `###`) to mirror the logical sections of the report, converting inline technical terms into backtick-wrapped `code spans`, migrating loose code excerpts into fenced ```` ``` ```` blocks with explicit language tags (`python`, `typescript`, `bash`) so that syntax highlighting renders correctly on GitHub, and unifying list styles (ordered vs. unordered) across parallel sections. AI also guided me in inserting relative-path image links and anchor references for the UML diagrams under `UML_files/` and the sprint review screenshots under `sprint_review/`, so that the rendered document on GitHub and the exported `Report Architecture and Design .docx` remain structurally equivalent.

More critically, AI assisted me in performing a **consistency cross-check between the written report and the actual codebase**. Because the report was authored in parallel with ongoing implementation, it was easy for narrative descriptions to drift away from the true state of the code—for example, referencing an outdated route path, a renamed service function, or an API field that had been refactored. I walked AI through the report section by section, pairing each technical claim against the corresponding source files (`flask-app/app/__init__.py`, `flask-app/app/services/prediction_service.py`, `flask-app/app/services/chat_service.py`, `react-app/src/api/chat.ts`, `scraper/main_scraper.py`, etc.), and AI flagged every mismatch: stale module paths, parameter signatures that no longer matched, SQLAlchemy model fields that had been renamed, and configuration keys that had moved into `.env`. For each discrepancy, AI proposed a minimal edit—either correcting the report to reflect the implemented code, or flagging the code path as genuinely deviating from the design intent so that I could decide whether to re-align the implementation. This bidirectional verification loop eliminated a substantial number of subtle inaccuracies before submission, ensuring that the final report is not merely a narrative artifact but a faithful, reviewable reflection of the shipped system.
